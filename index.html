TheaPantry is a smart food inventory and kitchen management application. Its primary goal is to reduce household food waste by tracking expiry dates, managing pantry stock across different storage zones (Fridge, Pantry, Freezer) and providing a visual overview of food consumption habits.

SDG 12 — Responsible Consumption and Production

Laboratory Modules 📂

Lab 1 (UI)

Built the TheaPantry home screen UI using Jetpack Compose — pantry cards with background images, an expiry badge, a bottom navigation bar and a "+ New List" button.

This was my first time building a full screen purely in Compose without XML, so figuring out how Box, Column and Row stack together took a bit of trial and error. Getting the card background image to sit behind the text at the right opacity was honestly more satisfying to crack than I expected.

Lab 2 (Interaction)

Added interactivity to the home screen — the pantry list now uses mutableStateListOf so users can add new custom pantry spaces on the spot, and it scrolls smoothly with LazyColumn.

State management clicked for me here, especially seeing how remember and mutableStateOf actually re-trigger the UI when values change. The "add new list" flow — hiding the text, showing the input field, then adding to the list — was small but it felt like the app finally had a pulse.

Lab 3 (Material Design)

Applied a custom Material 3 theme with full light/dark/dynamic color support, and added spring-animated expanding cards so each pantry entry shows a sustainability note when tapped.

Setting up the full color scheme token system in Theme.kt was tedious but worth it — swapping to dark mode and seeing everything adapt automatically was genuinely cool. The spring animation on the cards was a last-minute addition that ended up being one of my favorite small details in the whole project.

Lab 4 (Navigation)

Wired up multi-screen navigation using NavHost — the app now moves between Home, Shopping, Profile, and a Login screen, with the bottom bar hiding automatically on the login route.

Navigation arguments were tricky at first, especially making sure the back stack cleared properly after login so you can't press back into the auth screen. Once it worked cleanly end-to-end, it felt like the app went from a prototype to something that actually behaves like a real app.

Lab 5 (Room Database)

Set up a Room database with four tables — FoodItem, PantryItem, ShoppingItem, UserAccount — each with its own DAO for insert, update, delete, and Flow-based queries.

Room's DAO pattern made writing queries feel surprisingly clean once I got the hang of it. The singleton database instance and the Flow return type for getAllFoodItems took a bit to understand, but seeing the list update live whenever data changed was one of those moments where everything just made sense.

Featured Apps 🚀

Project 1: TheaPantry Base MVP

Problem Statement: Households often lose track of what's in their pantry — items expire unnoticed, groceries get bought twice, and there's no easy way to search or categorise what you have.

This was the first time everything came together: navigation, Room database, ViewModel, and a real detail screen where you can add food items with category, quantity, unit, and expiry date. Building the PantryDetailScreen with a search bar and category grouping took the most time, but it's the screen I'm most proud of in this project.

Project 2: Connected Cloud Ecosystem

Problem Statement: The offline-only app from Project 1 meant pantry data was stuck on one device, with no way to track food waste habits over time or discover recipes based on what you actually have.

Project 2 added a Waste Analytics screen that tracks consumed vs wasted items as a ratio, a Community Recipe screen that pulls from both Firestore and the Spoonacular API, and a camera screen for scanning items. It's a lot more moving parts than Project 1, and honestly debugging the Firestore reads while also juggling local Room data kept me on my toes the whole time.
